# 蕾丝图案网站技术方案报告（纯 Python 实施版）

## 一、项目概述

本项目建设一个主要供面料与服装设计人员使用的蕾丝图案参考展示与识图匹配工作台。首页展示 `pattern` 文件夹中的本月 Top 10 图案及一个统一更新时间；设计人员可上传客户参考图、查看模拟匹配进度，并依据临时文件名规则获得结果。匹配符合时进入成衣预览占位页；无法匹配或选择“不是这一款”时，可创建设计工单并登记客户信息。

当前阶段优先跑通完整业务闭环，不引入 Node.js、Vite 或真实 AI 模型，前后端均由 Python 项目统一提供。

## 二、软件与框架

### 1. 开发运行环境

- Windows 作为本地开发系统。
- Miniconda 提供 Python 3.11 解释器。
- 项目虚拟环境位于 `.venv`，用于隔离网站依赖。
- Visual Studio Code 可用于编辑和调试代码。
- Git 用于版本管理。
- Chrome 或 Edge 用于页面及移动端响应式调试。

### 2. 后端框架

- Python 3.11：主要开发语言。
- Flask 3：处理页面路由、模板渲染、图片上传、JSON 接口和静态资源。
- Werkzeug：由 Flask 使用，负责安全文件名及请求处理。
- Pillow：读取原始图案并生成 WebP 缩略图，减少首页流量。
- Python 标准库：负责目录扫描、文件名排序、JSON Lines 写入、时间处理和唯一编号生成。

### 3. 前端实现

- Jinja2：在服务端生成首页和预览页 HTML。
- HTML5：构建图案列表、上传区、进度区、结果确认和需求表单。
- CSS3：实现编辑部画册风格、蕾丝纹理氛围、动画和移动端布局。
- 原生 JavaScript：处理文件上传、模拟进度、接口请求、结果分支和表单提交。

该方案不依赖 Node.js 和前端构建工具，运行一个 Flask 服务即可同时提供前后端。

## 四、功能具体实现

### 1. Top 10 和统一更新时间

Flask 启动后不需要手工维护图案清单。访问首页时，`app.py` 扫描 `pattern` 文件夹，只读取 PNG、JPG、WEBP 和 BMP 图片，数字文件名按数值排序，最后取前 10 张。

统一更新时间取本批已展示图片中最新的文件修改时间，格式化为“年/月/日 时:分”，首页只展示一次。替换或更新任意图案后刷新页面即可看到新时间。

### 2. 图片展示优化

原始图案文件较大，若首页直接加载全部原图会影响打开速度。图片路由首次访问时使用 Pillow 将图片最长边缩小到 1200 像素，并生成质量为 84 的 WebP 缓存。缓存名称包含原图修改时间，原图更新后会自动产生新缓存，不需要手动清理。

### 3. 上传与临时匹配

前端通过 `FormData` 将文件发送至 `POST /api/match`。后端执行以下检查：

1. 是否选择文件。
2. 扩展名是否属于允许的图片类型。
3. 请求大小是否超过 12 MB。
4. 以随机编号保存文件，避免同名上传互相覆盖。
5. 读取用户上传时的原始文件名并与素材库文件名比较。

当前匹配规则为临时规则：忽略文件名大小写，除此以外必须同名。例如上传文件名为 `1.png` 时命中 `pattern/1.png`；上传 `lace.png` 时返回“暂未找到您想要的款式”。代码中已加入替换标记，后续接入图像向量算法时只需替换判断部分，接口响应结构可继续使用。

### 4. 匹配进度

上传开始后，页面隐藏上传区并显示环形动画、阶段文案、进度条和百分比。原生 JavaScript 用约 2 至 3 秒的分段定时器将进度推进到 94%，同时并行请求后端；两者都完成后进度到 100% 并展示结果。

该进度目前是用户体验模拟，不代表真实识别计算量。后续可改为任务编号加轮询接口，或者使用 Server-Sent Events/WebSocket 推送真实模型进度。

### 5. 设计人员确认与成衣预览

匹配成功后展示命中的图案，设计人员选择“符合我的需求”后才显示“点击预览成衣效果”按钮。按钮跳转到 `/preview`，当前页面显示占位文案和已选择的文件名，为未来成衣合成接口预留入口。

若设计人员选择“不是这一款”，则进入设计工单创建分支。

### 6. 设计工单与客户信息登记

没有命中或设计人员否认匹配结果时，页面先询问是否创建设计工单。确认创建后进入客户信息登记步骤，填写客户姓名和客户联系方式，前端发送 JSON 至 `POST /api/work-orders`。

后端校验字段后生成 `LS-日期-随机码` 格式的工单编号，并将记录按 JSON Lines 格式追加到 `data/design_work_orders.jsonl`。每条记录包含工单编号、客户姓名、客户联系方式、上传文件名、匹配状态、待处理状态、创建角色和创建时间。该方式适用于当前单机验证；正式上线且多人并发时建议换成 SQLite、PostgreSQL 或 MySQL。

后端同时校验业务来源，只有“无法匹配”或设计人员确认“不是这一款”的结果可以创建设计工单，避免已确认匹配的款式被重复转入设计流程。

## 五、接口设计

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| GET | `/` | 渲染 Top 10 与识图首页 |
| GET | `/pattern-images/<文件名>` | 返回缓存后的图案缩略图 |
| POST | `/api/match` | 接收上传并执行临时文件名匹配 |
| GET | `/preview?pattern=<文件名>` | 打开成衣预览占位页 |
| POST | `/api/work-orders` | 登记客户信息并创建设计工单 |

## 六、环境与代理配置

项目虚拟环境由 Miniconda 的 Python 3.11 创建。当前用户级包管理代理为：

```text
socks5h://127.0.0.1:3065
```

Conda 使用官方 `defaults` 频道，Pip 使用官方 `https://pypi.org/simple`。`socks5h` 会让域名解析也通过代理完成。此前失效的清华 Conda/Pip 镜像重定向已移除，避免 HTTPS 链路被重置。

## 七、本地运行与测试

在项目目录运行：

```powershell
.\.venv\Scripts\python.exe app.py
```

浏览器访问 `http://127.0.0.1:5000`。

自动化测试命令：

```powershell
.\.venv\Scripts\python.exe -m unittest discover -s tests -v
```

测试覆盖首页 Top 10、文件名匹配成功、匹配失败、缩略图生成、设计工单创建和成衣预览占位页。

## 八、Docker 测试与后续上线

当前本地开发不需要 Docker。需要在只有 Docker 的远程服务器测试时，可后续增加 `Dockerfile`，将项目和依赖打包为一个 Python 容器，并把 `pattern`、`uploads` 和 `data` 作为挂载目录。正式上线建议使用 Gunicorn/Waitress 承载 Flask，并由 Nginx 或云平台负责 HTTPS。

## 九、后续升级方向

1. 使用 OpenCV 或深度学习模型提取图案特征，以向量相似度替换文件名匹配。
2. 用真实后台任务状态替换延时进度。
3. 将设计工单迁移至数据库，并增加工单列表、分配、状态流转和检索功能。
4. 接入成衣模板合成或生成模型，实现真实效果预览。
5. 增加后台上传、排序和批次更新时间管理。

## 十、结论

纯 Python + Flask 方案已经覆盖当前完整业务链路，开发环境轻、启动方式简单，并保留了图像算法、数据库和成衣预览的升级接口。它适合先在本机快速验证，再封装为 Docker 容器部署到远程服务器。
